总判断

81 条真实命令里有 38 条通过、8 条失败、35 条超时。最危险的信号不是失败本身，而是 超时已经成为主导失效模式。这说明系统经常不是快速失败，也不是有效 fallback，而是一直挂到外层 150 秒 subprocess 总闸门为止。

如果只把问题归因为第三方接口不稳定，会误判优先级。更准确的判断是：上游不稳定一直存在，但 CLI 没有把这种不稳定约束在可控半径内。auto 对“慢但不报错”无能为力，请求层没有 provider 级 timeout，--limit 又只截输出不截请求，最终把可降级问题放大成整条命令超时。

从结构化统计看，efinance 明确指定时 50 条里有 35 条通过，说明 provider 本身并非整体不可用；真正拖垮体验的是 auto 兜底近乎失效，以及一批全量抓取命令没有时间边界。

一、脚本误用

stock holders latest-count

回测脚本传入 --date 20250331，但上游 [getter.py:924](/D:/github-project/efinance-cli/.venv/Lib/site-packages/efinance/stock/getter.py:924) 的 get_latest_holder_number() 使用 datetime.strptime(date, %Y-%m-%d)，要求横杠格式。所以 stderr 才会出现 time data '20250331' does not match format '%Y-%m-%d'。

stock leaderboard daily

脚本传入 --start-date 20250530 --end-date 20250530，而 [getter.py:1074](/D:/github-project/efinance-cli/.venv/Lib/site-packages/efinance/stock/getter.py:1074) 与 [getter.py:1077](/D:/github-project/efinance-cli/.venv/Lib/site-packages/efinance/stock/getter.py:1077) 同样要求 %Y-%m-%d。这说明脚本默认共享层日期写法可以通吃，但当前源码下不成立。

stock price live --market A_stock

共享 schema 在 [request_schema.py:80](/D:/github-project/efinance-cli/opentrade/request_schema.py:80) 把 A_stock 视为合法 market，但上游 [getter.py:322](/D:/github-project/efinance-cli/.venv/Lib/site-packages/efinance/stock/getter.py:322) 的 get_realtime_quotes(fs=...) 实际吃的是 FS_DICT key，第 426 到 428 行直接按 provider 语义取值。这不是普通样本笔误，而是共享抽象和 provider 原生抽象发生了语义错位。

二、产品缺陷 / 架构问题

auto 真实兜底几乎失效

[resolver.py:12](/D:/github-project/efinance-cli/opentrade/backends/resolver.py:12) 默认 backend 是 AUTO。[facade.py:65](/D:/github-project/efinance-cli/opentrade/facade.py:65) 的 _invoke_auto() 只有遇到异常才会切换候选 backend。如果前一个 backend 很慢但不抛异常，auto 就会一直卡住，直到外层 subprocess.run(timeout=150) 把整条命令杀掉。统计已经证明这一点：11 条 auto 用例只有 1 条真的落到了最终 backend，final_backend_counts 中有 10 条是 <none>。

请求层没有 provider 级 timeout

[base.py:121](/D:/github-project/efinance-cli/opentrade/backends/base.py:121) 会包一层调用重试，[retry_utils.py:38](/D:/github-project/efinance-cli/opentrade/retry_utils.py:38) 默认 max_retries=8。但追到的上游实现里，多处 session.get(...) 没有显式 timeout=，例如 [common/getter.py:55](/D:/github-project/efinance-cli/.venv/Lib/site-packages/efinance/common/getter.py:55)、[common/getter.py:152](/D:/github-project/efinance-cli/.venv/Lib/site-packages/efinance/common/getter.py:152)、[common/getter.py:472](/D:/github-project/efinance-cli/.venv/Lib/site-packages/efinance/common/getter.py:472)。当前真正的时间边界不是产品层，而是回测脚本的总超时闸门。

--limit 只裁剪渲染，不裁剪请求

[rendering.py:175](/D:/github-project/efinance-cli/opentrade/rendering.py:175) 的 maybe_limit() 只是对结果做 head()。这意味着系统会先请求、先标准化、先拼接，再在展示前裁掉几行。用户看到 --limit 5 会以为请求变轻，真实却只是“拿全量后少显示一点”，所以 futures catalog、market price live、futures price live 基本不减载。

stock profile 单标的走进多标的路径

共享 schema 把 --symbols 定义成多值，[request_schema.py:96](/D:/github-project/efinance-cli/opentrade/request_schema.py:96) 会把单值归一化为 list。随后上游 [getter.py:77](/D:/github-project/efinance-cli/.venv/Lib/site-packages/efinance/stock/getter.py:77) 的 get_base_info_muliti() 被触发，第 106 行还有 dropna(subset=[股票代码])。单标的请求被推进带并发和重试的多标的路径后，遇到远端断连就更脆弱。

三、上游 / 远端真实不稳定

yfinance 真实限流

quote price latest --backend yfinance 失败样本里出现 YFRateLimitError。从结果看，11 条 yfinance 用例 0 条通过、3 条失败、8 条超时，说明当前时点的真实可用性很差。

bond flow today 返回结构漂移

stderr 明确是 TypeError: 'bool' object is not subscriptable，说明上游返回结构已经不符合当前适配层的固定假设，不是普通网络抖动。

stock profile 远端断连

原始 stderr 出现 RemoteDisconnected 和 urllib3 MaxRetryError，说明远端连接被中断，且重试后仍未恢复稳定。

quote/common、futures、market 长时间挂起

[common/getter.py:27](/D:/github-project/efinance-cli/.venv/Lib/site-packages/efinance/common/getter.py:27) 的 get_realtime_quotes_by_fs() 会先拿第一页总量，再在第 77 行用 ThreadPoolExecutor 并发抓所有页，最后整合 DataFrame。[futures/getter.py:160](/D:/github-project/efinance-cli/.venv/Lib/site-packages/efinance/futures/getter.py:160) 的实时行情直接复用这条链路。这解释了为什么 quote price latest、quote trades、futures catalog、futures trades、market price live 更容易卡住：它们背后不是轻量点查，而是“先发现总量，再并发扫全页”的重查询。

重点命令逐条归因

search

默认路由和 auto 路由都容易长时间挂住，本质是慢 backend 没有被时间预算切走。

stock profile

远端断连叠加单标的走多标的路径，是上游问题和本地路径选择问题同时存在。

quote price latest

yfinance 限流叠加 auto 兜底失效，导致失败和超时同时出现。

quote trades

走 [common/getter.py:438](/D:/github-project/efinance-cli/.venv/Lib/site-packages/efinance/common/getter.py:438) 的重路径，慢调用没有边界。

futures catalog

核心不是 limit 太小，而是 limit 没有前移到请求层。

market price live

表面像轻量快照，实际是全市场抓取，--limit 仅影响展示。

修复优先级

P0：给 auto 加独立时间预算；给 provider / HTTP 层加显式 timeout。

P1：拆开共享 market 枚举和 provider fs 语义；为 stock profile 保留单标的路径；把 --limit 前移到请求层。

P2：修正回测样本，把日期格式和 market/fs 取值按真实 provider 分流。

排序依据很直接：先修“会把所有慢调用放大成整条命令超时”的问题，再修抽象错位，最后再清理回测样本，因为前两类直接决定真实用户能不能用。

对现有回测脚本的评价

[scripts/run_third_full_regression.py](/D:/github-project/efinance-cli/scripts/run_third_full_regression.py) 的覆盖设计本身并不差：expected_command_count=53、shared_auto_covered=11/11、explicit_backend_covered=26/26、business_option_covered=90/90。它的问题不在覆盖不够，而在样本值和当前产品边界没有完全贴齐，尤其是日期格式和 market/fs 语义差异没有在样本层面被区分开。换句话说，它作为压力暴露器是有价值的，但若要进一步升级为稳定回归门禁，必须先把“脚本误用”从“产品缺陷”中剥离出来。
