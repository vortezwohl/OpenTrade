# opentrade 命令自然化清单

## 1. 文档目标

本文档定义 `opentrade` 的目标命令体系，用于解决以下问题：

- 当前大量命令名直接照搬底层 Python API；
- `get-*`、`quote`、`bill`、`snapshot`、`fs` 等词带有明显实现味道；
- 同类能力命名粒度不一致，不利于猜测和记忆；
- `common`、`utils` 等开发者导向分组不适合作为默认用户入口；
- 参数已经开始语义化，但命令路径仍然不够自然。

本文档不是“现状说明”，而是“目标规范”。
默认前提如下：

- 不优先考虑历史兼容；
- 优先考虑直观、易记、易猜、低歧义；
- 命令命名要服务于 CLI 用户任务，而不是服务于底层函数命名。

---

## 2. 核心判断

当前 CLI 的主要问题，不是英文不英文，而是“命令名仍然是函数名思维”。

例如以下命令，对开发者可理解，但对普通 CLI 用户并不自然：

- `get-quote-history`
- `get-latest-quote`
- `get-quote-snapshot`
- `get-realtime-quotes`
- `get-history-bill`
- `get-deal-detail`
- `get-members`
- `get-belong-board`
- `get-realtime-quotes-by-fs`
- `search-quote-locally`

这些命令存在以下共同问题：

1. `get-` 前缀几乎不提供信息增量；
2. `quote` / `bill` / `snapshot` / `fs` 更像内部术语；
3. 命令名没有直接表达用户要完成的任务；
4. 同一领域下的命名骨架不统一；
5. 顶层分组混入了明显内部实现层。

结论：

- 参数统一化只能解决“命令该怎么填”；
- 命令自然化解决的是“用户该用哪个命令”；
- 后者决定 CLI 是否真的易学、易记、易传播。

---

## 3. 设计原则

## 3.1 命令名必须是任务名，不是函数名

应优先表达用户动作与业务对象，例如：

- `price history`
- `price latest`
- `flow today`
- `reports download`
- `constituents`

不应优先表达实现动作，例如：

- `get-quote-history`
- `get-history-bill`
- `get-pdf-reports`

## 3.2 去掉普遍性噪音词

以下词默认视为噪音词，应避免出现在最终主命令中：

- `get`
- `all`
- `by`
- `info`
- `data`

说明：

- CLI 本身就是“取数据”，`get` 没有区分度；
- “全部”通常应由参数表达，而不是硬编码进命令名；
- `by-fs` 这种实现路径不应直接暴露给普通用户；
- `info` 过于宽泛，应改成更具体的业务词。

## 3.3 同一领域必须共享同一命名骨架

例如行情类能力，应统一收敛到：

- `price latest`
- `price history`
- `price live`
- `price snapshot`

资金流类能力，应统一收敛到：

- `flow today`
- `flow history`

成交类能力，应统一收敛到：

- `trades`

基础资料类能力，应统一收敛到：

- `profile`

## 3.4 命令层级应体现业务结构

推荐命令结构：

```text
<domain> <topic> <action>
```

或在足够清晰时使用：

```text
<domain> <noun>
```

例如：

- `stock price history`
- `fund reports download`
- `bond flow today`
- `stock constituents`

不推荐继续大量使用：

```text
<domain> get-xxxx
```

## 3.5 面向普通用户的入口与高级入口分离

以下分组不应继续作为默认用户主入口：

- `common`
- `utils`

建议：

- 把通用搜索、解析能力提升到顶层；
- 把底层 `quote_id`/`fs` 类能力收敛到明确的高级命名空间；
- 普通用户优先走 `stock` / `fund` / `bond` / `futures` / `search` / `resolve`。

## 5. 顶层结构重组建议

## 5.1 推荐的用户可见顶层入口

```text
search
resolve
stock
fund
bond
futures
market
watch
```

说明：

- `search`：根据关键字搜索证券；
- `resolve`：把关键字或代码解析成内部标识；
- `market`：提供跨品类的市场扫描能力；
- `watch`：保留为统一刷新包装器。

## 5.2 不推荐继续暴露为主入口的分组

```text
common
utils
```

处理建议：

- `common` 的能力拆分到 `quote` 或 `market`，或者直接吸收到业务域命令；
- `utils` 中用户常用能力提升到顶层；
- 若必须保留，可转成隐藏/高级分组，不在默认帮助中主推。

## 6. 统一命名骨架

## 6.1 行情类

统一骨架：

```text
<domain> price latest
<domain> price history
<domain> price live
<domain> price snapshot
```

说明：

- `latest`：少量标的的最新价格/涨跌；
- `history`：K 线/净值历史；
- `live`：市场范围内的实时列表；
- `snapshot`：单标的当前完整快照。

## 6.2 资金流类

统一骨架：

```text
<domain> flow today
<domain> flow history
```

说明：

- `today` 表示日内分钟级；
- `history` 表示历史区间级流入流出。

## 6.3 成交类

统一骨架：

```text
<domain> trades
```

说明：

- `deal-detail` 统一改为 `trades`；
- 若未来存在不同粒度，再考虑 `trades today` / `trades history`。

## 6.4 基础资料类

统一骨架：

```text
<domain> profile
```

说明：

- `base-info` 不再保留；
- 当资料类型明确时再扩展，例如 `holders latest-count`。

## 6.5 下载/报告类

统一骨架：

```text
fund reports download
```

说明：

- 直接表达“下载基金报告”；
- 不把 `pdf` 这种文件格式放在主命令里；
- 文件格式由帮助文本和输出说明解释即可。

## 8. 明确废弃的命名模式

以下模式在目标体系中应视为禁用：

- `get-*`
- `*-info`
- `*-bill`
- `quote-*`（若该命令并非必须强调 `quote_id`）
- `realtime-*`
- `*-by-fs`
- `all-*`
- `search-quote-*`

说明：

- 废弃的是“命名模式”，不是单个词；
- 只要新命令仍然沿用这些模式，就说明还没有真正自然化。

## 9. 可操作的实施步骤

## 9.1 第一阶段：先定词汇，不急着改实现

先固定以下内容：

- 领域词典；
- 主题词典；
- 动作词典；
- 禁用词清单；
- 全量旧命令到新命令映射表。

没有这一步，后续实现很容易再次出现：

- 同类能力不同写法；
- 局部改名后仍旧 API 味很重；
- 一个词在不同地方表达不同概念。

## 9.2 第二阶段：重构注册层

实现层建议优先修改：

- `opentrade/registry.py`
- `opentrade/commands.py`

需要做的事：

- 为命令名引入显式 override 映射；
- 支持从 `get-quote-history` 这种函数名脱钩；
- 让帮助文本基于新命令路径组织；
- 把 `common` / `utils` 的对外可见性降级。

## 9.3 第三阶段：重构帮助文本与 README

必须同步更新：

- CLI `--help`
- [docs/cli-设计与使用说明.md](D:/github-project/opentrade/docs/cli-设计与使用说明.md)
- README 示例
- 回归测试中的命令调用样例

否则会出现：

- 命令名已经改了，但文档还是旧的；
- 新用户看帮助仍然无法建立正确心智。

## 9.4 第四阶段：分层迁移 `common` 与 `utils`

推荐迁移顺序：

1. 先把 `search-quote` 提升为顶层 `search`
2. 再把 `get-quote-id` 提升为 `resolve quote-id`
3. 再把 `get-realtime-quotes-by-fs` 转为 `market price live`
4. 最后决定 `quote` 高级命名空间是否仍需保留

原则：

- 优先迁移用户高频、可直观理解的入口；
- 最后再处理高级通用入口；
- 不要先把内部高级入口做得过重。

## 10. 重构优先级

## 10.1 第一优先级

- 去掉所有 `get-*`
- 行情类统一为 `price latest/history/live/snapshot`
- 资金流类统一为 `flow today/history`
- 成交类统一为 `trades`
- 基础资料类统一为 `profile`
- 顶层提升 `search`

## 10.2 第二优先级

- 收敛 `fund` 领域的 `nav` / `estimate` / `allocation` 命名
- 收敛 `stock` 领域的 `holders` / `ipo` / `leaderboard`
- 处理 `catalog` / `constituents` / `sector`

## 10.3 第三优先级

- 去用户可见化 `common`
- 去用户可见化 `utils`
- 建立 `quote` / `market` / `resolve` 的高级入口结构

## 11. 验收标准

命令自然化完成后，至少应满足以下标准：

1. 新用户看到命令名时，大致能猜出用途；
2. 同类能力的命令骨架高度一致；
3. 默认帮助中不再充斥 `get-*`；
4. 默认帮助中不再暴露明显底层实现术语；
5. 普通用户不需要先理解 `common` / `utils`；
6. 同一语义不会在不同模块下用不同词表达；
7. 从“我要查什么”到“我该敲什么命令”的映射足够短。

可用如下自检问题验收：

- 用户能否在不看源码的前提下猜出命令？
- 用户能否在看过一个行情命令后，举一反三猜出其他行情命令？
- 用户是否还需要理解 `quote`、`bill`、`fs` 这些内部词？
- 命令帮助是否更像产品说明，而不是 API 浏览器？

---

## 12. 一句话总结

最终目标不是把上游函数名从蛇形改成短横线，而是把 CLI 命令树重构成一套：

- 面向任务
- 面向用户
- 易猜易记
- 低歧义
- 可成体系扩展

的自然语言化命令规范。
