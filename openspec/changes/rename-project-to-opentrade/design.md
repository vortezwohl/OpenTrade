## Context

当前仓库的命名耦合面非常广，至少覆盖以下层级：

- 打包与发布层：`pyproject.toml` 中项目名为 `the-efinance-cli`，暴露脚本为 `efinance` / `efi`。
- 源码层：主包目录为 `efinance_cli`，`__init__.py`、`__main__.py`、`main.py` 和大量中文注释都显式引用旧产品名。
- 测试层：测试文件广泛通过 `from efinance_cli...` 导入，并在断言中检查旧命令名和旧展示文案。
- 文档层：`README.md`、`docs/`、历史 OpenSpec 变更、测试结论 HTML 中大量出现 `efinance-cli` 和 `efinance` 文案。

这不是单点替换，而是跨打包、源码、测试与主文档的横切重命名。该类变更存在明显破坏性，而且用户已明确要求彻底抹除旧命名，因此实现必须按一次性硬切换执行，不能再保留任何过渡层。

## Goals / Non-Goals

**Goals:**
- 明确项目对外名称统一为 `opentrade` / `Open Trade`。
- 明确 Python 包主路径统一为 `opentrade`。
- 明确 CLI 命令入口统一为 `opentrade` / `optr`。
- 明确旧命名必须被彻底移除，不保留兼容包、兼容命令或弃用提示。
- 让文档、测试、代码和发布元数据在迁移后保持一致。

**Non-Goals:**
- 本设计不重新规划业务命令树、后端能力或输出契约。
- 本设计不引入新的运行时依赖或新的架构层。
- 本设计不讨论品牌视觉、Logo 或站外营销资产。
- 本设计不保留任何旧命名的过渡窗口或双写支持。

## Decisions

### 决策 1：拆分为“品牌名 / 包名 / 命令名”三个 capability

**选择**：本次 rename 按 `project-brand-rename`、`python-package-rename`、`cli-command-rename` 三个 capability 拆分，而不是写成一个笼统 requirement。

**理由**：
- 这三个层级的验收方式不同：品牌名主要看文档和元数据，包名主要看导入和目录结构，命令名主要看 CLI 行为与兼容性。
- 拆开后更容易在实现阶段分批验证，避免一次性大改后难以定位问题。
- 任务拆分能直接映射到 OpenSpec tasks，符合本仓库的分步闭环要求。

**备选方案**：写成一个统一 capability。
放弃原因：会让 spec 过于宽泛，后续实现难以判断哪些变更属于必做、哪些是附带项。

### 决策 2：按一次性硬切换移除所有旧命名

**选择**：在 spec 中明确 `efinance-cli`、`the-efinance-cli`、`efinance_cli`、`efinance` 与 `efi` 都必须从主实现、打包入口、测试基线与主文档中彻底移除，不保留任何兼容入口。

**理由**：
- 用户已经显式要求“不要兼容，不要保留任何旧命名的兼容痕迹”。
- 如果仍然保留转发壳或弃用提示，会直接违背当前需求。
- 硬切换虽然破坏性更强，但实现边界更清晰，验证也更直接。

**备选方案**：保留短期兼容层。
放弃原因：与当前用户要求冲突。

### 决策 3：实现阶段优先做“机械可验证迁移”，避免顺手改行为

**选择**：实现阶段应优先使用最小行为面迁移策略：
- 先迁移目录、导入、脚本入口和文案；
- 再修复测试与文档；
- 不借机重构命令注册、执行管线或后端抽象。

**理由**：
- 这次变更的主风险来自命名面广，而不是业务语义复杂。
- 如果把 rename 和重构混在一起，验证失败时很难区分是命名迁移问题还是业务回归。
- 仓库已有较多测试覆盖，适合做一次偏机械化的全链路重命名并依赖回归兜底。

**备选方案**：借 rename 顺手整理目录和模块边界。
放弃原因：违反“精准修改”和“默认最小方案”原则。

### 决策 4：历史归档文档保持审计原样，但活跃路径彻底移除旧命名

**选择**：对 `openspec/changes/archive/` 中的历史文档不做回写；但当前主实现路径、当前活跃 change、README、主文档和测试基线中不得残留旧命名作为可执行或主路径身份。

**理由**：
- 历史归档文档属于审计记录，不宜为表面一致性大范围重写。
- 彻底“遗忘”在工程上应落在当前活跃路径，而不是伪造历史记录。

**备选方案**：一并全量替换 archive 文档中的旧命名。
放弃原因：收益低，且会污染历史审计记录。

## Risks / Trade-offs

- **[风险] 包目录重命名会导致大量导入路径一次性失效** → 通过全量替换主路径、运行测试和 `rg` 复查活跃引用缓解。
- **[风险] CLI 命令入口变更会直接破坏用户脚本与文档示例** → 通过同步更新 README / 主文档 / 回归测试，并接受这次 change 的纯 breaking 性质缓解。
- **[风险] 发布名和包名可能受 Python 包索引唯一性限制** → 实现前需要检查目标项目名 `opentrade` 是否作为发布坐标直接使用，若不可用则在 proposal 范围内补充发布命名折中方案。
- **[风险] 仓库中存在编码异常文档，批量替换时容易引入乱码或误改** → 实现阶段必须逐文件检查 UTF-8 编码结果，并优先改动当前活跃文档。
- **[权衡] 不保留兼容层会增加升级冲击** → 这是本次需求的显式约束，需通过文档与测试同步更新来控制工程风险，而不是通过保留旧入口缓解。

## Migration Plan

1. 先确认发布坐标、包目录名和 CLI 命令名的最终映射。
2. 迁移源码目录与入口配置，使 `opentrade` 成为唯一主路径。
3. 删除旧脚本入口和旧包路径，不保留任何兼容壳。
4. 同步更新测试、README、主文档和必要的 OpenSpec 活跃文档。
5. 运行定向与全量验证，确认命令入口、导入路径、构建与测试均通过。
